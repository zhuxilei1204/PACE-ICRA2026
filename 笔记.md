当前根因短记：

1. A3 仍在复用 T1 的中间目标几何。具体包括：身体目标 x = ball_future_x - 0.1；身体目标 y = ball_future_y + 0.6；击球/预测 x 被限制在 -1.6 附近；无效球 fallback 身体点是 [-1.80, 0.3, body_height]；actor 观测里也写死了 ball_prediction 的 [-0.1, +0.6] 偏移。这些在 T1 上成立，但未必适配 A3 的身高、臂长、球拍安装和舒适击球区。

2. A3 要重新标定自己的几何假设：默认击球姿态下，记录 base/pelvis 到球拍击球中心的 offset；球拍正面法向；舒适击球高度范围；击球前身体目标位置；击球窗口内球拍速度方向；击球后可恢复姿态。

3. A3 还缺少挥拍/恢复动作先验。先不做复杂人类动捕，优先做工程化关键帧：ready -> small backswing -> impact -> follow-through -> recovery，并作为软奖励/参考轨迹，不要硬写死动作。

4. 下一步不只是调 reward weight，而是做 A3 专用 target_base_pos、target_racket_pos、target_racket_vel、target_paddle_normal、strike_window，并记录这些误差到 TensorBoard 和可视化。

学习小结：
初始姿态是矫正的第一步，接下来是调整适合A3的几何假设，有这个几何假设，才可以说把一个把球打到对面的任务折算成可以学习的中间命令，球未来会到哪里，机器人身体应该站在哪里，球拍应该出现在身体的哪个相对位置，身体应该往哪个方向移动。
PACE的流程是球当前状态 -> 物理预测 ball_future_pose-> 根据几何假设计算 robot_future_pos / robot_future_vel-> actor 观测到相对身体目标-> critic/reward 用 robot_future_pos / robot_future_vel 打分-> policy 输出关节目标-> PD 执行动作。

就是说球拍击球点相对于机器人base的位置，如果不修正，如果球拍已经接近球路了，但是算法却会认为身体在另外一个位置。这样的话会认为这样移动是错误的，reward会错误的进行奖励，这样球拍奖励和身体奖励会相互冲突。

几何假设是如何计算出 robot_future_pos / robot_future_vel：
先预测球未来适合击球的位置 ball_future_pose
再根据“球拍击球点相对机器人 base 的几何 offset”
反推出机器人身体应该站在哪里 robot_future_pos
再根据当前位置到目标位置的差，算一个期望身体速度 robot_future_vel
比如pace代码：
robot_future_pos_x = ball_future_x - 0.1
robot_future_pos_y = ball_future_y - paddle_y_offset
robot_future_pos_z = body_height

robot_future_vel = clamp((robot_future_pos - robot_pos) * 4, min=-7, max=7)

目前采用FK来计算A3的几何假设，之后如果出现以下情况，需要重新计算：
1.如果 A3 右臂 ready 姿态变了，这个 offset 要重算。
2.如果球拍 URDF 安装位置变了，要重算。
3.如果真实机器人球拍安装和仿真 URDF 不一致，sim2real 前要重新标定。
4.它只说明“当前姿态下球拍中心相对 base 在哪里”，不是完整挥拍轨迹。
5.它主要修正 robot_future_pos / robot_future_vel 的身体目标，不等于已经解决拍面法向、击球速度、击球窗口。


对于actor 观测到相对身体目标-> critic/reward 用 robot_future_pos / robot_future_vel 打分这一步的理解：
可以理解为一个是actor看到未来要往那里去的提示，而reward来判断actor是否真的往那边去了。
